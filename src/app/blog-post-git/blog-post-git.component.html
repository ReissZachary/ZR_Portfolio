What is git

Git is an open source version control system. What does that mean? Well, open source simply means that anyone...yes..even you.. can contribute to the project. This is done by checking out the code, writing some super awesome new feature, submitting your code to be reviewed, then pushing your changes to production. Version control systems main purpose is to maintain and track code changes over time. Version control systems, like git, store all the changes made to code in a database, along with other data like who made the change, and what type of chages were made. Git is one of those systems and using something like git has many benefits.
Consider the following..

  • Like mentioned above, git track code changes over a period of time. This means that if something goes wrong, a record of the changes and events which lead up to the problem will be available.
  • Git allows you (and your team) to store different versions of their project. So, as soon as a new release is ready, it can be stored alongside a name, a description of the release, and any other relevant metadata (such as the name of the author, and so on).
  • Git is very efficient and very easy for beginners to pick up. Cloning, merging, pushing, and committing can be done with nice, clean, easy to remember syntax.
  • Git is used by many many developers. This means that finding docs/support is very easy
  • Git includes something called branching which lets you, along with other team members, work on different parts of the code without disrupting one another

Git Jargon

When first learning git, you will most likely hear terms that you don't know what they mean. Some of the most common are "commit", "push", "branch", and "repository". Lets define some of these before looking at an example:

repository - Also known as a 'repo'. The best way to put this is that git uses 'repository' to refer to your project. So think of a repo simply as your project. A repo not only stores your code but it keeps track of change history as well

commit: - A commit is a saved change made to a repository. Whenever you make a change to your code, you should commit it to the Git repository. This adds a record of the change you have made to the changelog of your repository, to which you can refer back in the future.

push: - pushing your code means that you are taking your code (on your machine) in a local repo and transferring it to a remote repo somewhere on a different machine.

branch: - Branches are independent versions of the code stored in a repo which allow developers to make multiple changes to a codebase without affecting the main copy of the code.

Practice!

For this practice session, I created a new .NET Core console application.

To initial a new git repo for a project, make sure you are in that project directory and run git init.

Now that git is initialized, lets make some changes. I went ahead and changed my Program.cs file to instead of saying "Hello World" it now says "Hellow Wonderful World". Since I changed my code I should tell git to track and stage it to get ready for commit. To do this I run git add (filename). I think it is worth mentioning that git add . with the . after the add will do the same thing. The . means to add all changed files to the repo.

To ensure we are tracking and have added the right files, you can run git status. This tells you are the files being tracked currently. This is what it would look like after we made the change mentioned above and added that file to git isnt that nice??!

Once you've added all the changed files you want, you are ready to commit! To commit your changes, simply run the command git commit -m 'your message'. Here, the '-m' means that the following text is going to be the commit message. It is alway best to have clear commit messages. Lets see this in our demo Now that you've commited your code, you are ready to push the committed changes into production/master. The command for this is git push.
